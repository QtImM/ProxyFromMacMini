# VPN与Docker隔离SOP

## 1. 目的

这份 SOP 专门解决两个最容易把项目搞乱的问题：

1. `mac mini` 通过 `VPN/代理` 使用 Codex
2. `mac mini` 现有 Docker 任务不能受影响

## 2. 风险总览

如果不做隔离，最容易出的问题有：

1. `cloudflared` 走进代理链，隧道不稳定
2. API 或 Redis 被错误代理，内部通信异常
3. Docker 容器继承代理变量，行为不可控
4. 新系统端口、网络、项目名撞上已有 Docker 任务
5. 任务中心抢占资源，拖慢或打挂原有任务

## 3. 代理隔离原则

### 3.1 必须分成两层

第一层是基础设施层：

1. `cloudflared`
2. API
3. Redis
4. 普通 worker

第二层是代理敏感任务层：

1. `codex_run`
2. 其他确实需要外部代理的任务

### 3.2 基础设施层默认规则

基础设施层默认规则必须是：

1. 不继承 Codex 代理变量
2. 不依赖系统代理
3. 内部通信直连本地地址或内部网络

### 3.3 任务层默认规则

任务层默认规则必须是：

1. 普通任务不注入代理
2. 只有确实需要外网代理的任务显式注入代理
3. 代理注入必须带 `NO_PROXY`

## 4. NO_PROXY 建议

无论你最后怎么实现，`NO_PROXY` 至少要覆盖：

1. `127.0.0.1`
2. `localhost`
3. `::1`
4. `redis`
5. API 与 worker 的内部服务名

目的只有一个：  
防止本机内部调用也被错误发到代理链里。

## 5. Docker 隔离原则

### 5.1 命名隔离

新任务中心必须使用独立命名空间：

1. 独立 Compose 项目名
2. 独立 Docker 网络名
3. 独立卷名前缀

### 5.2 端口隔离

必须保证：

1. 新服务只占用新规划端口
2. 不复用现有业务容器端口
3. Redis 不对公网开放
4. worker 不对公网开放

### 5.3 资源隔离

建议从一开始就考虑：

1. 单 worker 单槽位
2. 资源使用可观测
3. 重任务最后接入

## 6. 实际执行 SOP

### 6.1 部署前

先完成：

1. 盘点所有 Docker 容器
2. 盘点所有 Docker 网络
3. 盘点所有 Docker 卷
4. 盘点端口占用
5. 盘点当前代理变量与 VPN 路径

### 6.2 部署中

严格遵循：

1. 先起基础设施层
2. 不接代理任务
3. 本地链路验证通过
4. 再接 Cloudflare Tunnel
5. Tunnel 稳定后再接 `codex_run`

### 6.3 部署后

重点观察：

1. Tunnel 是否掉线
2. API 是否超时
3. Redis 是否异常
4. 现有 Docker 任务是否异常重启
5. 现有 Docker 任务资源是否被抢占
6. `codex_run` 是否因代理不稳定失败

## 7. 不允许的做法

下面这些做法不建议用：

1. 把 `mac mini` 整机全局流量都混到一个代理里，然后不做区分
2. 让基础设施层和 `codex_run` 使用同一套代理注入逻辑
3. 不盘点现有 Docker 环境就直接起新项目
4. 一开始就四槽全开并接入重任务
5. 直接暴露 `mac mini` 公网端口代替 Tunnel

## 8. 推荐的最稳路径

最稳的路径仍然是：

1. 先盘点
2. 再做单槽基础设施验证
3. 再接 Tunnel
4. 再扩四槽
5. 最后再接入 `codex_run`

这样做的好处是，一旦出问题，你能快速判断：

1. 是 Tunnel 问题
2. 是内部基础设施问题
3. 是代理问题
4. 还是现有 Docker 环境冲突问题
