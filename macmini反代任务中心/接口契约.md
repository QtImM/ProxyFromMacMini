# 接口契约

## 1. 设计原则

这份契约只定义第一版最小可用接口，不扩张。

原则如下：

1. 长任务必须异步
2. 客户端只提交和查询
3. 服务端统一决定槽位
4. `task_type` 必须白名单化
5. 不提供任意命令执行入口

## 2. 核心接口

### 2.1 `POST /tasks`

用途：提交新任务。

客户端建议字段：

1. `task_type`
2. `payload`
3. `source_machine`
4. `requested_by`
5. `idempotency_key`
6. `priority`

说明：

1. `task_type` 只能是白名单中的值
2. `payload` 结构由 `task_type` 决定
3. `source_machine` 固定来自 `win-a` `win-b` `win-c` `win-d`
4. `idempotency_key` 用于避免重复提交

返回字段建议：

1. `task_id`
2. `status`
3. `slot`
4. `queue_position`
5. `created_at`

### 2.2 `GET /tasks/{task_id}`

用途：查询任务状态。

返回字段建议：

1. `task_id`
2. `task_type`
3. `status`
4. `slot`
5. `created_at`
6. `started_at`
7. `finished_at`
8. `attempt_count`
9. `error_message`
10. `result_ready`

### 2.3 `GET /tasks/{task_id}/result`

用途：获取任务结果。

返回建议分两类：

1. 结构化结果
2. 文件结果索引

结构化结果示例字段：

1. `task_id`
2. `status`
3. `result_json`

文件结果示例字段：

1. `task_id`
2. `status`
3. `result_files`
4. `summary`

### 2.4 `GET /slots`

用途：查看 4 个槽位状态。

返回字段建议：

1. `slot-a`
2. `slot-b`
3. `slot-c`
4. `slot-d`

每个槽位下建议返回：

1. `state`
2. `current_task_id`
3. `queue_size`
4. `source_machine`

### 2.5 `POST /tasks/{task_id}/cancel`

用途：取消任务。

第一版约束：

1. 只允许取消 `queued`
2. 不处理 `running` 强制终止

## 3. 状态机

统一状态只保留这 5 个：

1. `queued`
2. `running`
3. `succeeded`
4. `failed`
5. `canceled`

不建议第一版引入更多状态。

## 4. 槽位路由规则

服务端统一路由，客户端不负责选槽位。

固定规则：

1. `win-a -> slot-a`
2. `win-b -> slot-b`
3. `win-c -> slot-c`
4. `win-d -> slot-d`

第一版不启用自动借槽。

## 5. 任务类型约束

第一版建议任务类型只开放少数白名单：

1. `health_check`
2. `script_run`
3. `collect_materials`
4. `generate_outline`
5. `codex_run`

每个任务类型都必须明确：

1. 输入字段
2. 超时限制
3. 是否需要代理
4. 输出格式

## 6. 幂等约束

幂等处理要求如下：

1. 同一个 `idempotency_key` 重复提交时，优先返回已存在任务
2. 不允许因为网络抖动导致重复执行
3. 幂等作用域建议至少覆盖 `source_machine + task_type + idempotency_key`

## 7. 失败处理约束

第一版失败处理建议收敛：

1. 失败要有 `error_message`
2. 失败要保留日志索引
3. 第一版可不自动重试，或最多一次重试

## 8. 安全约束

必须加的约束：

1. 客户端请求必须鉴权
2. 不允许开放 shell 原始执行接口
3. 不允许 Redis 暴露公网
4. 不允许 worker 内部接口暴露公网
