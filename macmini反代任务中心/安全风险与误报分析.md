# 安全风险与误报分析

## 1. 这份文档为什么必须补

当前文档已经覆盖了两类风险：

1. 不影响 `mac mini` 里的 Docker 服务
2. 不让 `Codex` 的代理环境污染任务中心基础设施

但你给出的真实案例说明，还缺少第三类风险分析：

**我们的平台即使是正常操作，也可能在现有系统里表现得像攻击。**

这类风险不一定会把 Docker 打挂，但会带来：

1. 安全告警误报
2. 运维误判
3. API 被误认为遭扫目录或撞库
4. 正常自动化行为与异常行为无法区分
5. 真攻击发生时，日志上也分不清是不是我们自己的任务中心

所以这份文档的目标是：

1. 识别我们的服务可能制造的“像攻击”的行为
2. 区分“误报风险”和“真实安全风险”
3. 在设计阶段就给出规避方案

## 2. 先给结论

### 2.1 是否考虑到了“不影响 Docker 服务”

**考虑到了，但目前主要考虑的是基础设施层面。**

也就是：

1. 端口不冲突
2. 网络不冲突
3. Compose 项目不冲突
4. 卷不冲突
5. 资源不抢占

这些在现有文档里已经覆盖了。

### 2.2 还没完全补齐的是什么

**还没完全补齐的是应用层和安全审计层面的影响。**

也就是：

1. 我们的正常请求会不会像扫描
2. 我们的自动化行为会不会像攻击
3. 现有系统会不会把我们认成陌生外部流量
4. 出现 4xx/5xx 时，能不能一眼看出是“平台正常任务”还是“异常探测”

这才是你刚才这个案例真正提醒我们的重点。

## 3. 必须分层思考风险

我们要把风险分成 4 层，而不是只看 Docker 会不会挂。

### 3.1 第 1 层：基础设施层风险

这层已经覆盖较多，主要是：

1. 端口冲突
2. Docker 网络冲突
3. 卷污染
4. CPU / 内存 / 磁盘抢占
5. Tunnel 与代理链相互影响

这一层的目标是：

**别把现有服务搞崩。**

### 3.2 第 2 层：流量行为层风险

这一层是你案例里最重要的。

如果任务中心发出的流量具有下面这些特征，就很容易被误判为攻击：

1. `curl` 或 `Python urllib` 风格的请求头
2. 短时间密集请求多个 API 路径
3. 自动化访问不存在或内部管理路径
4. 连续 POST、PUT、DELETE 操作
5. 高频查询状态接口

这一层的目标是：

**别让正常平台操作看起来像扫站、撞库或脚本攻击。**

### 3.3 第 3 层：权限与越权风险

即使是我们自己的任务，如果设计不好，也可能真的越权：

1. 平台可以访问本不该碰的内部接口
2. 任务 payload 被滥用，转成任意 API 调用器
3. 平台拿到了高权限 token 或内部 key
4. 平台能创建、修改、删除现有系统中的对象

这一层的目标是：

**别把任务中心做成一个高权限攻击跳板。**

### 3.4 第 4 层：审计与归因风险

就算所有操作都是合法的，如果日志上看不出“是谁触发的、为什么触发的”，也会很危险。

典型问题：

1. 日志里只有 `curl/8.x`
2. 无法区分是人手操作、平台任务还是攻击脚本
3. 无法追到具体是哪一台机器、哪个用户、哪个任务 ID
4. 任务完成后没有审计闭环

这一层的目标是：

**出了问题时，必须能快速归因。**

## 4. 按你的案例倒推，我们的平台会带来哪些具体风险

下面按最现实的顺序来分析。

### 风险 1：任务中心的请求会被误判成扫描或攻击

如果平台做这些事，就很像异常流量：

1. 用 `curl` 直接打内部 API
2. 为了“探测能力”访问很多路径
3. 自动重试太频繁
4. 出错后批量重放请求

这和你案例里的现象非常像：

1. `curl` 用户代理
2. 多路径探测
3. 批量访问管理 API
4. 产生 500 或连续 200

#### 对策

必须要求平台遵守：

1. 不做目录探测式访问
2. 不扫描未知路径
3. 每个任务只访问明确白名单接口
4. 不允许“自己发现接口”
5. 自动重试要限次、限速

### 风险 2：平台流量来源不清，会被当成外部陌生攻击者

如果我们的平台流量从这些位置发出：

1. VPN 出口
2. 代理出口
3. Cloudflare 回源
4. 其他公网 NAT

那在现有系统看来，它可能就是：

1. Google Cloud
2. 某个云 IP
3. 非内部地址
4. 看起来像互联网上来的脚本

#### 对策

必须明确平台的流量身份：

1. 固定出口策略
2. 固定调用入口
3. 固定 `User-Agent`
4. 固定 `X-Task-Source`
5. 固定 `X-Task-Id`
6. 固定 `X-Requested-By`

目标是：

**让现有系统一眼看出来这是“任务中心合法调用”，不是匿名 curl。**

### 风险 3：平台可能真的触发现有系统中的危险动作

这不是误报，而是真实风险。

如果平台能直接访问已有业务系统，并带着高权限身份，就可能：

1. 创建对象
2. 修改配置
3. 删除资源
4. 消耗模型额度
5. 触发审计事件

你给出的案例里就出现了：

1. 创建 Agent
2. 修改 Agent
3. 删除 Agent
4. 调用 chat
5. 消耗模型额度

#### 对策

平台与现有业务系统交互时，必须：

1. 用专门的低权限账号
2. 用专门的任务身份，不共用 admin
3. 每类任务限定允许调用的 API
4. 平台侧禁止任意 URL / 任意方法调用
5. 平台侧禁止直接持有长期高权限 token

### 风险 4：我们的平台可能掩盖真正的攻击

如果平台本身也会发出很多自动化请求，而日志上又看不出归属，那么真实攻击发生时，你会很难判断：

1. 这是平台正常任务
2. 还是外部攻击
3. 还是平台被劫持

#### 对策

必须做最小审计标准：

1. 每个任务有 `task_id`
2. 每次外部系统调用带 `task_id`
3. 带 `source_machine`
4. 带 `requested_by`
5. 带 `task_type`
6. 任务中心本地保留调用日志索引

目标是：

**现有系统看到请求时，能反查到任务中心里的具体任务。**

### 风险 5：高频轮询和失败重试会压现有系统

即使没有攻击意图，也可能造成业务压力：

1. 四个人同时提交任务
2. 每个任务频繁轮询
3. 自动失败重试
4. 调用下游系统时没有节流

这会带来的问题：

1. API 压力上升
2. 触发风控或误报
3. 把现有服务日志刷满
4. 掩盖真正异常

#### 对策

平台必须内建：

1. 轮询频率上限
2. 下游调用频率上限
3. 每任务最大重试次数
4. 对已有业务系统的节流策略

### 风险 6：平台一旦拿到内部 key，会变成高危跳板

你给出的案例里有一个很典型的问题：

`X-Internal-Key` 硬编码在网关配置里。

如果未来任务中心被设计成依赖类似内部 key：

1. key 泄露风险极高
2. 平台等于拿着通行证访问内部系统
3. 一旦任务中心被滥用，就不是误报，是实打实越权

#### 对策

必须坚持：

1. 不把高权限内部 key 暴露给普通任务
2. 不把任务中心设计成“万能内网代理”
3. 不把现有业务系统 admin 身份直接接进任务中心

## 5. 对我们这个项目的结论

### 5.1 目前文档有没有考虑“不影响 Docker 服务”

有，而且考虑得比较完整，但主要是：

1. 基础设施资源层面
2. Docker 隔离层面
3. 代理隔离层面

### 5.2 目前文档还需要补的重点

现在必须把下面这 4 条也升级成硬约束：

1. 任务中心不得做未知路径探测
2. 任务中心不得以匿名 curl 风格访问已有系统
3. 任务中心访问下游系统必须带可归因标识
4. 任务中心不得直接持有高权限业务 admin 身份

## 6. 应该新增的硬规则

从现在开始，这个项目应新增以下规则：

### 规则 1：平台只允许白名单任务，不允许自由探测

平台不允许：

1. 扫路径
2. 猜接口
3. 探测后台页面
4. 枚举未知资源

### 规则 2：平台调用外部系统必须可识别

必须至少带：

1. 统一 `User-Agent`
2. `X-Task-Id`
3. `X-Task-Type`
4. `X-Requested-By`
5. `X-Source-Machine`

### 规则 3：平台与现有业务系统之间必须使用专用低权限身份

不能用：

1. 现有 admin 账号
2. 共享 root 级 token
3. 硬编码内部绕过 key

### 规则 4：平台对已有系统的访问必须有限流

必须限制：

1. 单任务调用频率
2. 单槽位调用频率
3. 重试次数
4. 失败后的退避间隔

### 规则 5：平台日志要能和现有系统日志对上

必须做到：

1. 平台有任务日志
2. 下游系统日志能看到任务标识
3. 出现异常时能双向对账

## 7. 对部署顺序的影响

基于这些风险，部署顺序还要再加一层要求：

1. 单槽上线前，先定义统一请求标识规范
2. Tunnel 接入前，先定义平台身份
3. 接 `codex_run` 前，先确认不会把匿名脚本流量打到现有业务系统
4. 对接任何已有业务系统前，先准备专用低权限账号

## 8. 新的阶段验收要求

在原有阶段验收之外，新增下面几个 gate：

### Gate A：平台身份可识别

过关标准：

1. 现有系统日志能看出是任务中心调用
2. 能看出是哪一个任务
3. 能看出是哪个人或哪台机器发起

### Gate B：不会被误判为扫描器

过关标准：

1. 平台不访问未知路径
2. 平台不进行目录探测
3. 平台不使用匿名默认 curl 行为模式

### Gate C：即使误报，也能快速自证

过关标准：

1. 平台本地日志保留完整任务链路
2. 下游系统日志带任务标识
3. 两边能快速对账

## 9. 一句话结论

这套项目现在不能只追求：

“不影响 `mac mini` 的 Docker 服务。”

还必须同时追求：

“不把正常任务中心流量做成像攻击，也不让任务中心真的变成高权限跳板。”
