# 部署手册

## 1. 目的

这份手册用于把 `4 台机器 -> mac mini -> 执行任务中心` 的方案，转换成真正可以上机执行的部署顺序。

核心目标：

1. 不打断现有 Docker 任务
2. 不污染 `mac mini` 上现有 Codex 代理环境
3. 先搭基础设施，再接复杂任务
4. 从单槽验证扩展到四槽

## 2. 总体部署原则

部署时必须坚持下面 5 条原则：

1. 先盘点，再部署
2. 先基础设施，再执行槽位
3. 先单槽验证，再扩到 4 槽
4. 先本地链路验证，再接 Cloudflare Tunnel
5. 任何时候都不直接改动现有 Docker 任务

## 3. 阶段一：环境盘点

正式开始前，先完成这几件事：

1. 列出现有 Docker 容器
2. 列出现有 Docker Compose 项目
3. 列出现有端口占用
4. 列出现有 Docker 网络
5. 列出现有 Docker 卷
6. 确认 `mac mini` 当前代理方式
7. 确认 `Codex` 是否依赖系统级代理、Shell 变量代理，还是 VPN 客户端
8. 确认机器 CPU、内存、磁盘余量

这一步的目的不是修改任何东西，而是建立基线。

## 4. 阶段二：端口与命名规划

在真正部署前，先固定命名规则。

建议统一如下：

1. 项目名：`taskhub`
2. Docker 网络名：`taskhub_net`
3. 数据卷前缀：`taskhub_*`
4. API 本地端口：`18080`
5. Redis 本地端口：`16379`
6. 可选监控页端口：`19090`

这样做的目的：

1. 防止和现有 Compose 项目重名
2. 防止误连现有容器网络
3. 排障时一眼识别哪些资源属于任务中心

## 5. 阶段三：基础设施上线顺序

建议严格按下面顺序进行：

1. 先启动 Redis
2. 再启动 API
3. 再启动 `slot-a`
4. 本地验证任务提交和状态查询
5. 再接入 `Cloudflare Tunnel`
6. 验证外部访问 API
7. 然后再启动 `slot-b`
8. 再启动 `slot-c`
9. 最后启动 `slot-d`

不要一开始就四槽全开。

## 6. 阶段四：链路验证顺序

### 6.1 本地链路验证

先验证：

1. API 是否本地可访问
2. API 是否能连接 Redis
3. 单槽 worker 是否能消费任务
4. 任务状态是否会从 `queued` 变为 `running`
5. 完成后是否能更新为 `succeeded` 或 `failed`

### 6.2 外部链路验证

本地验证通过后，再验证：

1. `Cloudflare Tunnel` 是否能稳定转发到 API
2. 外部机器是否能访问 `/health`
3. 外部机器是否能提交任务
4. 外部机器是否能查询任务状态

### 6.3 并发验证

最后再做：

1. `win-a`、`win-b`、`win-c`、`win-d` 同时各提交一个任务
2. 检查是否分别进入 `slot-a` 到 `slot-d`
3. 检查互不抢占
4. 检查一个槽位故障不会拖垮其他槽位

## 7. Cloudflare Tunnel 接入原则

Tunnel 接入时要注意：

1. Tunnel 只指向本地 API
2. 不暴露 Redis
3. 不暴露内部 worker 端口
4. API 服务最好仍然只监听本地地址
5. 通过 Cloudflare 提供统一域名入口

推荐思路是：

`公网域名 -> Cloudflare Tunnel -> 本地 API`

而不是：

`公网域名 -> 直接进 mac mini 暴露端口`

## 8. 执行层上线原则

每个槽位都当成独立车道处理。

上线顺序建议：

1. 先验证 `slot-a`
2. 再复制同样模式到 `slot-b`
3. 再复制到 `slot-c`
4. 最后复制到 `slot-d`

每个槽位都应该有：

1. 自己的队列
2. 自己的运行状态
3. 自己的日志标识

## 9. 任务类型上线顺序

不要第一天就让所有任务类型都进系统。

建议顺序：

1. `health_check`
2. `script_run`
3. `collect_materials`
4. `codex_run`

把 `codex_run` 放到最后，是因为它最容易和代理问题缠在一起。

## 10. 验证通过标准

只有满足下面条件，才算第一版部署通过：

1. 本地 API 可用
2. Tunnel 外部访问可用
3. 单槽任务可提交流转完成
4. 四槽可同时各跑一个任务
5. 普通任务不受代理污染
6. `codex_run` 可按需使用代理
7. 现有 Docker 任务没有异常重启、端口冲突或资源抢占报警

## 11. 上线后观察点

上线后前几轮运行必须重点看：

1. API 响应时间
2. Redis 稳定性
3. Tunnel 连通性
4. 单槽平均任务时长
5. 资源占用峰值
6. 现有 Docker 任务是否被影响
7. `codex_run` 在代理条件下是否稳定

## 12. 当前最稳的执行建议

如果要尽快开始真正实施，我建议照下面节奏来：

1. 今天先完成盘点
2. 明天先做单槽验证
3. 单槽稳定后再扩成四槽
4. 最后才接入正式 `codex_run`
