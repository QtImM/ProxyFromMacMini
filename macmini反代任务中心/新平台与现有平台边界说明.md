# 新平台与现有平台边界说明

## 1. 这份文档的目的

这份文档专门用来解决一个非常容易混淆的问题：

**我们现在要做的新平台，不等于 `mac mini` 上已经在运行的现有平台。**

这两者必须严格区分。

如果边界不清，后面很容易出现这些问题：

1. 把现有平台的风险误当成新平台内部设计问题
2. 把新平台设计成要接管现有平台
3. 排障时分不清是谁的问题
4. 部署时错误地共享资源、账号或网络

所以这份文档的目标是：

1. 把两个平台的身份分开
2. 把共存关系说清楚
3. 把影响边界钉死
4. 给后续部署和验收一个统一口径

## 2. 两个平台分别是什么

### 2.1 现有平台

现有平台是：

1. 已经在 `mac mini` 上运行的既有业务系统
2. 有自己的 Docker 容器、网关、接口、认证、日志和安全状态
3. 已经在真实使用中
4. 有自己的历史问题和安全风险

这套平台不是本项目要重新设计的对象。

### 2.2 新平台

新平台是：

1. 我们现在规划的 `macmini反代任务中心`
2. 一个新的统一入口与任务调度层
3. 负责让 4 个人通过统一入口向 `mac mini` 提交任务
4. 有自己的 API、队列、槽位、日志和运行规则

这套平台是本项目真正要建设的对象。

## 3. 两个平台的关系

### 3.1 不是同一平台

必须明确：

1. 新平台不是现有平台的重构版本
2. 新平台不是现有平台的替代物
3. 新平台不是现有平台的子模块
4. 新平台不是现有平台的管理后台

### 3.2 是同机共存关系

当前我们要处理的是：

1. 两个平台可能运行在同一台 `mac mini`
2. 两个平台可能共用一部分宿主机资源
3. 两个平台可能共用一条外网出路
4. 两个平台可能共用同一 Docker 宿主环境

所以核心问题不是“它们是不是一个平台”，而是：

**新平台如何与现有平台安全共存。**

## 4. 新平台不能默认做的事

为了避免混淆，新平台默认不应承担以下职责：

1. 不负责接管现有平台的认证
2. 不负责接管现有平台的网关
3. 不负责接管现有平台的日志系统
4. 不负责接管现有平台的管理员身份
5. 不负责自动管理现有平台里的对象

也就是说：

**新平台首先是一个独立平台，不是现有平台的控制台。**

## 5. 两个平台共享什么，不共享什么

### 5.1 可能共享的东西

只在宿主环境层面，可能共享：

1. 同一台 `mac mini`
2. 同一套 Docker 引擎
3. 同一条公网网络出口
4. 同一台机器的 CPU / 内存 / 磁盘

### 5.2 绝不能默认共享的东西

必须默认隔离：

1. Docker Compose 项目名
2. Docker 网络
3. 数据卷
4. 服务端口
5. 管理员账号
6. 内部 key
7. 高权限 token
8. 日志命名空间
9. 任务队列

## 6. 正确的风险表述方式

以后讨论风险时，应该这样表达：

### 6.1 正确说法

1. 新平台会不会影响现有平台的 Docker 容器
2. 新平台会不会抢占现有平台资源
3. 新平台流量会不会被现有平台误判为异常
4. 新平台是否需要调用现有平台
5. 如果要调用，调用边界如何限制

### 6.2 不建议的说法

1. 我们的平台内部怎么访问自己后台
2. 我们的平台怎么登录自己系统
3. 平台里的 admin 是否共享

这些说法会把两个平台说成一个系统，容易把边界搞乱。

## 7. 对部署设计的直接影响

这个边界明确后，部署设计必须满足：

1. 新平台用独立端口
2. 新平台用独立 Docker 资源名
3. 新平台默认不接触现有平台内部接口
4. 新平台即使部署失败，也不能拖垮现有平台
5. 新平台的代理配置不能污染现有平台

## 8. 如果未来两个平台需要交互，应该怎么定义

当前默认前提是：

**新平台可以与现有平台共存，但不默认耦合。**

如果未来需要交互，必须额外补一层“平台间接口边界”，而不是直接共享内部身份。

正确做法应是：

1. 明确交互目的
2. 明确交互接口
3. 明确交互账号
4. 明确交互频率限制
5. 明确审计字段

而不是：

1. 直接复用现有平台 admin
2. 直接复用现有平台内部 key
3. 直接让新平台随意打现有平台所有 API

## 9. 对安全分析的修正

基于这个边界，安全分析应该修正成：

1. 新平台自己的安全风险
2. 新平台对现有平台的影响风险
3. 现有平台是否会把新平台误判为异常流量
4. 如果未来发生平台间调用，如何避免越权

而不是把现有平台的安全事件直接当成新平台内部事件。

## 10. 对阶段验收的影响

后续阶段验收时，要新增一个固定问题：

**本阶段讨论的问题，是新平台内部问题，还是新平台对现有平台的影响问题？**

每次都要先分清这一点。

否则很容易：

1. 在错误对象上做优化
2. 在错误平台上加限制
3. 错误归因

## 11. 一句话结论

一句话钉死这个边界：

**我们现在做的是一个部署在同一台 `mac mini` 上的新平台，它需要与现有平台安全共存，但它不是现有平台本身。**
