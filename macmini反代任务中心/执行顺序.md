# 执行顺序

## 1. 这份文档的作用

如果你现在只想知道“先干什么，后干什么”，就看这一份。  
它把整个项目压缩成严格顺序，避免文档很多但不知道从哪开始。

## 2. 总执行顺序

整个项目按下面顺序推进：

1. 先读 `规划文件`
2. 再读 `阶段验收计划`
3. 然后执行 `环境盘点记录模板`
4. 用 `预检清单` 检查盘点是否完整
5. 用 `VPN与Docker隔离SOP` 确认风险隔离方案
6. 用 `接口契约` 冻结接口边界
7. 用 `部署手册` 确定实际上机顺序
8. 用 `实施计划` 进入真正的实现阶段

## 3. 第一阶段现在就该做什么

你现在不该先写代码，也不该先上服务。

现在第一动作是：

1. 打开 [环境盘点记录模板.md](</C:\Users\Tim\Desktop\东方人物档案短视频执行计划\macmini反代任务中心\环境盘点记录模板.md>)
2. 在 `mac mini` 上逐项盘点真实环境
3. 把 Docker、端口、代理、Codex 依赖方式、资源余量都记录下来

这一阶段的目标不是实现系统，而是拿到真实环境事实。

## 4. 第二阶段做什么

盘点完成后，做这几件事：

1. 回看 [预检清单.md](</C:\Users\Tim\Desktop\东方人物档案短视频执行计划\macmini反代任务中心\预检清单.md>)
2. 确认盘点项是否都能回答
3. 把未决风险单独列出来
4. 判断是否允许进入“单槽验证准备”

这一阶段的目标是判断：  
当前 `mac mini` 真实环境，是否允许继续推进。

## 5. 第三阶段做什么

当盘点和预检都通过后，再做架构冻结：

1. 确认入口就是 `Cloudflare Tunnel`
2. 确认执行层就是 `4 个固定槽位`
3. 确认第一版不自动借槽
4. 确认长任务全部异步
5. 确认 `codex_run` 最后接入

这一阶段的目标是：  
把系统结构冻结，避免实施时反复改大方向。

## 6. 第四阶段做什么

架构冻结后，再进入最小闭环定义：

1. 单槽先行
2. 只做最小 API 链路
3. 先不做四槽全开
4. 先不接复杂代理任务

这一阶段的目标是：  
定义第一版最小可用实现，不让项目起步过重。

## 7. 第五阶段做什么

最小闭环明确后，才进入真正实施：

1. 先做单槽
2. 再做本地链路
3. 再接 Tunnel
4. 再扩四槽
5. 最后接入 `codex_run`

这一阶段的目标是：  
真正把文档转成可运行系统。

## 8. 最关键的顺序纪律

下面这个顺序不能乱：

1. 盘点先于实现
2. 单槽先于四槽
3. 本地链路先于公网入口
4. 普通任务先于代理任务
5. 基础设施稳定先于 `codex_run`

## 9. 当前你的起点

如果按现在的文档状态判断，你已经完成的是：

1. 方案定义
2. 文档框架
3. 风险边界设计

你还没开始的，是：

1. `mac mini` 真实环境盘点
2. 阶段 1 验收
3. 单槽实施准备

## 10. 一句话版本

一句话执行顺序就是：

`先盘点 mac mini -> 再过预检 -> 再冻结架构 -> 再做单槽 -> 再接 Tunnel -> 再扩四槽 -> 最后接入 codex_run`
