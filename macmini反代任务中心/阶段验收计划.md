# 阶段验收计划

## 1. 目的

这份文档把整个项目拆成一组可以逐阶段验收的 gate。  
只要你按顺序执行这些阶段，并且每一阶段都满足退出标准，就可以把当前这套方案收敛成一个完整、可实施、可验证的项目。

这份文档解决的不是“怎么写代码”，而是：

1. 每一阶段先做什么
2. 每一阶段产出什么
3. 每一阶段怎么判断过关
4. 哪些问题没解决前不能进下一阶段

## 2. 总体推进原则

整个项目必须遵守这 6 条推进原则：

1. 先确认边界，再做实现
2. 先做低风险验证，再做高风险接入
3. 先保证不影响现有环境，再追求功能完整
4. 先打通最小闭环，再扩展四槽并发
5. 先解决代理与隔离，再接入 Codex 实际任务
6. 任何阶段不过关，都不进入下一阶段

## 4. 阶段 0：项目定义冻结

### 4.1 目标

确认项目边界，避免边做边变成另一个系统。

### 4.2 本阶段输入

1. `规划文件`
2. 你已经确认的关键约束

### 4.3 本阶段必须确认的内容

1. 入口采用 `Cloudflare Tunnel`
2. 执行中心统一放在 `mac mini`
3. 并发模型为 `4 个固定槽位`
4. 默认一人一槽，不自动借槽
5. `mac mini` 使用 `VPN/代理` 跑 Codex
6. 新系统不能影响现有 Docker 任务
7. 长任务必须异步
8. 新平台不是现有平台本身
9. 新平台默认不接管现有平台的认证、网关或 `admin` 身份

### 4.4 本阶段产出物

1. 目标冻结
2. 约束冻结
3. 技术方向冻结

### 4.5 本阶段退出标准

下面问题都已经没有歧义，才能过关：

1. 为什么不用 Tailscale 作为主方案
2. 为什么不是同步 API
3. 为什么不是公共并发池
4. 为什么必须做代理与 Docker 隔离

## 5. 阶段 1：环境基线验收

### 5.1 目标

在不改动现有环境的前提下，把 `mac mini` 现状摸清。

### 5.2 本阶段输入

1. `预检清单`
2. `VPN与Docker隔离SOP`

### 5.3 本阶段必须完成的动作

1. 盘点现有 Docker 容器
2. 盘点现有 Compose 项目
3. 盘点端口占用
4. 盘点 Docker 网络与卷
5. 盘点当前代理方式
6. 盘点 `Codex` 的代理依赖方式
7. 盘点资源余量

### 5.4 本阶段产出物

建议至少形成一份环境记录，内容包括：

1. 容器列表
2. 端口列表
3. 网络列表
4. 卷列表
5. 代理路径说明
6. 风险点摘要

### 5.5 本阶段退出标准

只有下面问题都答得出来，阶段 1 才算过：

1. 当前最不能碰的 Docker 容器有哪些
2. `18080`、`16379`、`19090` 是否可用
3. 代理变量目前是系统级、shell 级还是客户端级
4. Docker 是否可能继承这些代理变量
5. 机器是否有余量承接 4 槽位

## 6. 阶段 2：架构验收

### 6.1 目标

把系统结构冻结到足够细，让后续实现不会反复改大方向。

### 6.2 本阶段输入

1. `规划文件`
2. `接口契约`
3. `部署手册`

### 6.3 本阶段必须确认的内容

1. 统一入口链路
2. 4 槽位映射关系
3. 核心接口列表
4. 状态机
5. 任务白名单
6. Redis 与结果目录分工
7. Cloudflare Tunnel 只指向 API

### 6.4 本阶段产出物

1. 确认版接口定义
2. 确认版槽位模型
3. 确认版部署拓扑

### 6.5 本阶段退出标准

只有下面问题没有争议，阶段 2 才算过：

1. 客户端是不是只和 API 通信
2. 服务端是不是统一路由到槽位
3. Redis 是不是只做状态和队列，不做结果文件
4. `codex_run` 是不是最后接入

## 7. 阶段 3：隔离策略验收

### 7.1 目标

在真正实施前，把最危险的两个问题先过关：

1. 代理污染
2. Docker 冲撞

### 7.2 本阶段输入

1. `VPN与Docker隔离SOP`
2. `预检清单`

### 7.3 本阶段必须确认的策略

1. 基础设施层默认不继承代理
2. 普通任务默认不继承代理
3. 只有 `codex_run` 按需注入代理
4. `NO_PROXY` 覆盖本地通信
5. 新系统使用独立 Compose 项目名
6. 新系统使用独立网络和卷
7. 新系统不直接复用现有容器网络
8. 新平台默认不扫描现有平台的未知路径
9. 新平台如需访问现有平台，必须带可归因标识

### 7.4 本阶段产出物

1. 代理隔离规则
2. Docker 隔离规则
3. 资源隔离原则

### 7.5 本阶段退出标准

下面任一问题答不上来，都不能进入下一阶段：

1. `cloudflared` 会不会走进代理链
2. Redis 会不会误走代理
3. API 会不会继承 Codex 代理
4. 新系统会不会和现有 Docker 项目重名

## 8. 阶段 4：最小闭环验收

### 8.1 目标

定义第一版最小闭环，防止一开始做得过大。

### 8.2 最小闭环定义

第一版最小闭环只要求做到：

1. 可提交任务
2. 可查状态
3. 可查结果
4. 只验证单槽
5. 只验证本地链路

### 8.3 本阶段产出物

1. 单槽验证范围
2. 第一版任务类型范围
3. 第一版不做清单

### 8.4 本阶段退出标准

必须明确“第一版不做什么”，至少包括：

1. 不做自动借槽
2. 不做高可用
3. 不做任意命令执行
4. 不做复杂权限系统
5. 不做多机 worker

## 9. 阶段 5：单槽验证验收

### 9.1 目标

在不接四槽、不接复杂任务的前提下，跑通一个完整车道。

### 9.2 本阶段验证范围

1. 单个槽位
2. 单个白名单任务
3. 本地 API
4. 本地队列
5. 本地状态流转

### 9.3 本阶段必须验证的能力

1. 任务提交成功
2. 返回 `task_id`
3. 状态能从 `queued` 变 `running`
4. 最终能到 `succeeded` 或 `failed`
5. 日志与结果有索引

### 9.4 本阶段退出标准

只有下面都满足，单槽才算过关：

1. 流程跑通至少一轮
2. 状态流转符合接口契约
3. 不影响现有 Docker 任务
4. 不依赖代理也能正常跑普通任务

## 10. 阶段 6：Tunnel 接入验收

### 10.1 目标

在单槽跑通后，再把入口从本地扩到外部机器。

### 10.2 本阶段必须验证的能力

1. Tunnel 可用
2. 域名可用
3. 外部机器可访问 API
4. 外部机器可提交任务
5. 外部机器可查询状态

### 10.3 本阶段退出标准

只有下面都满足，Tunnel 阶段才算通过：

1. `/health` 外部可访问
2. 任务提交后能回到本地执行链路
3. Redis 与 worker 没有对外暴露

## 11. 阶段 7：四槽扩展验收

### 11.1 目标

把单槽模式稳定复制到四槽。

### 11.2 本阶段必须验证的能力

1. `win-a -> slot-a`
2. `win-b -> slot-b`
3. `win-c -> slot-c`
4. `win-d -> slot-d`
5. 四个槽位互不抢占
6. 一个槽位阻塞不影响其他槽位

### 11.3 本阶段退出标准

只有下面都满足，四槽阶段才算通过：

1. 四台机器都能独立提交任务
2. 每台机器都落到固定槽位
3. 不出现公共池抢占
4. 不出现槽位串扰

## 12. 阶段 8：Codex 代理任务验收

### 12.1 目标

最后再接入真正最敏感的 `codex_run`。

### 12.2 本阶段必须验证的能力

1. 普通任务不带代理
2. `codex_run` 按需带代理
3. `NO_PROXY` 不影响本地链路
4. Tunnel 不受代理影响
5. Docker 基础设施层不受代理污染

### 12.3 本阶段退出标准

下面全部满足，才算真正完成第一版：

1. `codex_run` 可稳定执行
2. 普通任务与代理任务互不污染
3. 现有 Docker 任务无异常
4. 四槽并发场景下仍稳定

## 13. 阶段 9：上线准备验收

### 13.1 目标

确认项目已经从“方案”进入“可正式实施”状态。

### 13.2 本阶段产出物

应至少具备：

1. 确认版架构文档
2. 确认版接口契约
3. 部署顺序手册
4. 预检清单
5. 风险隔离 SOP
6. 新旧平台边界说明
7. 安全风险与误报分析
8. 阶段 gate 计划

### 13.3 本阶段退出标准

当下面这句话成立时，项目就算准备完毕：

“即使换一个执行者，只看这个目录下的全部 `.md`，也能知道项目目标、边界、接口、部署顺序、预检方法、隔离策略和阶段验收标准。”
