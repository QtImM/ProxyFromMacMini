# macmini反代任务中心规划文件

## 1. 目标

构建一套面向 `4 台机器 -> mac mini -> 执行任务中心` 的统一入口系统。

目标如下：

1. 四台客户端机器只负责发起任务请求与查询结果。
2. 所有真正执行动作统一在 `mac mini` 上完成。
3. 公网入口不直接暴露 `mac mini`，而是通过 `Cloudflare Tunnel` 暴露域名入口。
4. 执行层采用 `4 个固定槽位`，默认一人一槽，互不抢占。
5. 设计时必须考虑 `mac mini` 本机通过 `VPN/代理` 使用 Codex。
6. 设计时必须考虑 `mac mini` 现有 Docker 任务不能被新系统干扰。

## 2. 推荐总体方案

推荐技术组合：

1. `cloudflared`
2. `FastAPI`
3. `Redis`
4. `RQ` 或等价轻量 worker 队列
5. `Docker Compose`

推荐逻辑结构：

1. 入口层：`Cloudflare Tunnel`
2. 控制层：`API 服务`
3. 状态与队列层：`Redis`
4. 执行层：`4 个固定 worker 槽位`
5. 结果层：本地文件目录与结构化结果

## 3. 系统拓扑

### 3.1 外部访问链路

1. Windows A/B/C/D 调用统一域名，例如 `https://api.example.com`
2. `Cloudflare Tunnel` 将请求转发到 `mac mini` 本地 API 端口
3. API 服务进行鉴权、校验、任务入队
4. 对应槽位 worker 拉取任务执行
5. 结果写回 Redis 与本地结果目录
6. 客户端轮询状态并读取结果

### 3.2 本机服务链路

建议 `mac mini` 本机采用以下服务端口规划：

1. `127.0.0.1:18080`：任务中心 API
2. `127.0.0.1:16379`：Redis
3. `127.0.0.1:19090`：内部管理或健康检查页面，可选

原则：

1. 默认仅绑定 `127.0.0.1`
2. 不额外暴露本地端口到局域网
3. 由 `cloudflared` 负责对外映射

## 4. 并发与槽位模型

### 4.1 固定 4 槽位

执行层固定为：

1. `slot-a`
2. `slot-b`
3. `slot-c`
4. `slot-d`

映射关系默认固定：

1. `win-a -> slot-a`
2. `win-b -> slot-b`
3. `win-c -> slot-c`
4. `win-d -> slot-d`

### 4.2 调度规则

1. 每个槽位同一时间只跑 `1` 个任务
2. 每个槽位各自维护独立队列
3. 一个槽位阻塞，不影响其他槽位
4. 第一版不启用自动借槽
5. 总并发上限为 `4`

### 4.3 为什么不用公共并发池

不建议做成“一个公共队列 + 并发 4”的原因：

1. 容易出现某一台机器占满全部执行资源
2. 使用者预期不稳定
3. 排障时不容易判断是哪一路堵住
4. 不符合你当前“一人一个槽”的使用习惯

## 5. API 设计

### 5.1 核心接口

1. `POST /tasks`
2. `GET /tasks/{task_id}`
3. `GET /tasks/{task_id}/result`
4. `GET /slots`
5. `POST /tasks/{task_id}/cancel`

### 5.2 提交任务字段

建议 `POST /tasks` 字段如下：

1. `task_type`
2. `payload`
3. `source_machine`
4. `requested_by`
5. `idempotency_key`
6. `priority`

### 5.3 返回字段

建议返回：

1. `task_id`
2. `status`
3. `slot`
4. `queue_position`
5. `created_at`

### 5.4 状态机

统一任务状态：

1. `queued`
2. `running`
3. `succeeded`
4. `failed`
5. `canceled`

## 6. 任务类型约束

第一版不要开放“任意命令执行”。

必须做成白名单任务类型，例如：

1. `codex_run`
2. `script_run`
3. `collect_materials`
4. `generate_outline`
5. `health_check`

每个 `task_type` 单独定义自己的 `payload schema`。

## 7. Redis 与结果目录设计

### 7.1 Redis 键结构

建议至少包含：

1. `task:{task_id}`：任务详情
2. `queue:slot-a`
3. `queue:slot-b`
4. `queue:slot-c`
5. `queue:slot-d`
6. `slot:slot-a:current`
7. `slot:slot-b:current`
8. `slot:slot-c:current`
9. `slot:slot-d:current`

### 7.2 本地结果目录

建议目录结构：

1. `runs/{task_id}/input.json`
2. `runs/{task_id}/stdout.log`
3. `runs/{task_id}/stderr.log`
4. `runs/{task_id}/result.json`
5. `runs/{task_id}/artifacts/`

## 8. 新增重点约束一：mac mini 通过 VPN/代理使用 Codex

这是本项目必须提前处理的核心约束之一。

### 8.1 风险点

如果 `mac mini` 上 Codex 依赖 VPN 或代理，最容易出现的问题有：

1. 新系统容器继承全局代理变量，导致内部服务互访异常
2. Docker 容器流量误走 VPN，造成 Redis 或本地 API 通信变慢或失败
3. `cloudflared` 被错误地放进代理链，导致隧道不稳定
4. Codex 所需代理和任务中心所需本地通信混在一起，排障困难

### 8.2 设计原则

原则必须固定：

1. `Codex` 的代理使用，与任务中心内部通信隔离
2. `cloudflared` 不默认继承 Codex 的代理环境变量
3. `API`、`Redis`、`worker` 之间的本地通信优先直连 `127.0.0.1`
4. 仅让真正需要外网代理的执行任务显式携带代理环境

### 8.3 推荐做法

推荐分层处理：

1. 任务中心基础设施层
   - `cloudflared`
   - `API`
   - `Redis`
   - `worker`

这层默认不走全局代理。

2. 任务执行层
   - 只有 `codex_run` 或明确需要代理访问外网的任务，才注入代理环境变量

### 8.4 代理变量建议

对需要代理的任务，在 worker 子进程级别按需注入：

1. `HTTP_PROXY`
2. `HTTPS_PROXY`
3. `ALL_PROXY`
4. `NO_PROXY`

其中 `NO_PROXY` 至少要覆盖：

1. `127.0.0.1`
2. `localhost`
3. `::1`
4. `redis`
5. 本机内部任务中心服务名

### 8.5 规划中的检查项

在正式上线前，必须执行以下检查：

1. `cloudflared` 在无代理变量下可稳定连通
2. API 服务在无代理变量下可访问 Redis
3. worker 在无代理变量下可消费队列
4. `codex_run` 任务在显式代理变量下可正常访问外部服务
5. 同一台机器上，代理任务与非代理任务互不污染环境

## 9. 新增重点约束二：不能影响 mac mini 现有 Docker 任务

这是第二个必须提前纳入规划的硬约束。

### 9.1 风险点

当前 `mac mini` Docker 已经跑着现有任务，新系统可能造成：

1. 端口冲突
2. 容器名冲突
3. 网络桥接冲突
4. CPU 和内存抢占
5. 磁盘写入放大
6. 全局环境变量污染
7. Docker Compose 项目名重叠

### 9.2 设计原则

必须遵循：

1. 新系统使用独立 `compose project name`
2. 新系统使用独立网络名
3. 新系统使用独立卷名
4. 不改动已有容器、不重启已有容器
5. 所有上线前检查都先做只读检查

### 9.3 上线前只读检查清单

部署前先检查，不允许直接改动现有 Docker 环境：

1. `docker ps`
2. `docker compose ls`
3. `docker network ls`
4. `docker volume ls`
5. 检查现有容器占用端口
6. 检查现有容器是否使用全局代理变量
7. 检查当前机器 CPU、内存、磁盘余量

### 9.4 资源隔离建议

新系统容器建议设置明确资源边界：

1. API：低资源，优先稳定
2. Redis：中低资源，避免过高内存上限
3. worker：按槽位分别限制资源

建议第一版：

1. 每个 worker 单独容器或单独进程
2. 每个 worker 配置 CPU 与内存软限制
3. 高负载任务必须写日志并可回查

### 9.5 端口与网络规避策略

建议：

1. 内部服务只绑定回环地址或内部 Docker 网络
2. 不使用现有业务可能占用的常见端口
3. 通过 `cloudflared -> API` 单点转发，不直接映射一组公网端口

### 9.6 规划中的检查与绕开计划

必须把以下动作列为部署前 gate：

1. 盘点现有 Docker 容器、端口、网络、卷
2. 为新项目指定独立项目名，例如 `taskhub`
3. 为新项目指定独立 network，例如 `taskhub_net`
4. 为新项目指定独立 volumes
5. 明确新系统所有端口仅本地使用
6. 上线前先 `dry-run` 配置核对
7. 上线后先启动基础设施，再启动单个 worker 验证
8. 确认无冲突后再启动全部 4 个槽位

## 10. 推荐部署顺序

### 阶段 1：环境盘点

1. 检查现有 Docker 任务
2. 检查现有代理与 VPN 方式
3. 检查本机资源余量
4. 确认本地端口规划

### 阶段 2：基础设施落地

1. 部署 Redis
2. 部署 API
3. 部署单 worker
4. 配置 `cloudflared`
5. 打通一条端到端链路

### 阶段 3：槽位扩展

1. 增加 `slot-b`
2. 增加 `slot-c`
3. 增加 `slot-d`
4. 验证 4 路并发互不干扰

### 阶段 4：代理任务验证

1. 验证普通任务不走代理
2. 验证 `codex_run` 显式走代理
3. 验证 `NO_PROXY` 覆盖本地内部通信
4. 验证开启代理不会破坏 Cloudflare Tunnel 和 Redis

## 11. 第一版不做的事情

为保证先稳定落地，第一版先不做：

1. 自动借槽
2. 动态扩容
3. 多机 worker
4. 任意命令执行
5. 复杂权限系统
6. 自动回调通知
7. 图形化管理后台

## 12. 最终建议结论

当前最适合你的路线仍然是：

`cloudflared + FastAPI + Redis + 4 固定槽位 worker`

但在实施时必须额外加上两条硬约束：

1. `Codex` 的 VPN/代理能力与任务中心基础设施通信隔离
2. 对现有 Docker 任务先盘点、再绕开、后部署，不能直接碰现有运行环境

如果后续开始实施，第一步不是写 worker，而是先做：

1. `mac mini` 现有 Docker 环境盘点
2. `VPN/代理` 环境变量与网络路径盘点
3. 端口与 Compose 隔离方案确认
